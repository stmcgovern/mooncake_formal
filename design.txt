Mooncake KV cache disaggregated serving

What we formalize and why:

  Mooncake disaggregates KV cache from GPU compute in LLM serving.
  Prefill nodes produce KV blocks; these are stored in a distributed
  pool (DRAM, SSD, CXL); decode nodes fetch them on demand via RDMA.
  The core insight: trade cheap storage for expensive GPU recomputation.

  Strip away the transport layer and the scheduler — what remains is a
  distributed object store with a lifecycle protocol.  The objects are
  KV cache blocks.  The lifecycle is: produce → transfer → store →
  serve → evict.  The store manages two interlocking state machines:
  replica status (per-object) and segment status (per-container).

  The core correctness question: do the protocol gates compose safely?
  Each gate is trivial alone — readable iff COMPLETE, allocatable iff
  SegOk, protected iff pinned or referenced.  But their COMPOSITION
  has subtle interleavings that testing can't enumerate: can you drain
  a segment while a transfer is writing to a replica on it?  Can you
  evict an object that another transfer is about to read?

  We prove: (1) each state machine has no self-transitions and terminal
  states are absorbing, (2) eviction protection is coherent (pinned OR
  referenced OR in-progress → protected), (3) segment operations respect
  status gates, (4) transfer slice counting is monotonic, (5) the
  protocol gates compose into a single global invariant.

What we model and what we leave out:

  MODEL (high verification value):
  - Replica status state machine (INITIALIZED→PROCESSING→COMPLETE→REMOVED, +FAILED)
  - Segment status lifecycle (OK→DRAINING→DRAINED→UNMOUNTING)
  - Eviction protection (pin levels, ref counting, status gates)
  - Transfer task lifecycle (PENDING→ACTIVE→COMPLETED/FAILED)
  - Slice counting invariant (done_slices ≤ total_slices, monotonic)
  - Segment capacity bookkeeping (used ≤ capacity)
  - Object metadata and placement safety (allocate only on SegOk)
  - Drain/unmount preconditions (empty + drained before unmount)

  LEAVE OUT (low verification value or too coupled to internals):
  - RDMA/NVLink transport layer details (hardware-specific, not protocol-level)
  - Allocator implementations (CachelibBuffer, OffsetBuffer — memory management internals)
  - Standby/HA state machine (STOPPED→CONNECTING→...→PROMOTED — complex but orthogonal)
  - OpLog replication and snapshot mechanics (persistence layer, not core protocol)
  - P2P store shard distribution (Go codebase, etcd metadata — different language/paradigm)
  - EP (expert parallelism) — separate concern, formalized in deepep_formal
  - File storage backend, io_uring — I/O implementation details
  - Eviction strategy implementations (LRU/FIFO ordering — algorithm, not safety)
  - Lease TTL renewal mechanics (timing-dependent, hard to model usefully)
  - IBGDA/RoCE queue pair setup (hardware initialization)

MODELING DECISIONS
==================

Non-obvious choices that affect multiple files.

  1. Replica status without UNDEFINED: Source has 6 states (UNDEFINED,
     INITIALIZED, PROCESSING, COMPLETE, FAILED, REMOVED).  We drop
     UNDEFINED — it represents uninitialized memory, not a protocol
     state.  A replica enters our model at INITIALIZED.
     WHY: UNDEFINED has no meaningful transitions and adds noise to
     every exhaustive case analysis.

  2. Weights as nat, not floats: Eviction priority, transfer progress,
     and object sizes are all nat.  We don't model floating-point
     watermark ratios (e.g. high_watermark_ratio=0.95).
     WHY: The safety properties (eviction protection, capacity bounds)
     are structural, not numerical.  Nat arithmetic keeps proofs
     decidable.

  3. Pin levels as a 3-element type, not nat: Source uses implicit
     priority; we make it explicit with Unpinned < SoftPinned < HardPinned
     and a decidable ordering.
     WHY: Makes eviction protection predicates compositional — we can
     state "hard-pinned implies protected" as a standalone lemma.

  4. Transfer slices as counts, not buffers: Source tracks actual data
     buffers per slice with variant descriptors (RDMA, memcpy, file).
     We track only (done_slices, total_slices) counts.
     WHY: The safety property is monotonicity (done ≤ total, done
     increases), not data content.  Buffer types are transport-layer
     concerns we exclude.

  5. Segment status without UNDEFINED: Same rationale as replica — a
     segment enters our model at SegOk (mounted and available).
     WHY: Keeps segment_transition inductive clean.

  6. Eviction protection is a bool predicate, not a lock: Source uses
     atomic refcnt + weak_ptr + scoped access.  We model the logical
     predicate: protected iff hard-pinned OR ref_count > 0 OR not-yet-
     readable.
     WHY: The invariant is "protected objects are not evicted", not
     "the lock is held correctly".  The latter is a concurrency concern
     we don't model.

HEADLINE THEOREMS
=================

  Layer 1 — State machines (Core.v):
  1. replica_no_self_transition — no replica status is its own successor
  2. segment_no_self_transition — no segment status is its own successor
  3. no_transition_from_removed — REMOVED is terminal for replicas
  4. no_transition_from_unmounting — UNMOUNTING is terminal for segments
  5. processing_not_evictable — in-progress writes are always protected
  6. hard_pinned_protected — hard-pinned objects are always protected
  7. active_readers_protected — objects with ref_count > 0 are protected
  8. readable_is_complete — readability implies COMPLETE status

  Layer 2 — Segment operations (SegmentManager.v):
  9. place_requires_ok — placement requires SegOk segment
  10. empty_drained_safe — empty + drained = safe to unmount

  Layer 2 — Transfer protocol (TransferProtocol.v):
  11. create_task_slice_inv — new tasks satisfy slice invariant
  12. complete_slice_preserves_inv — slice completion is monotonic
  13. transfer_no_self_transition — no transfer status loops
  14. task_finished_iff — task_finished reflects terminal status

  Layer 3 — Drain protocol (DrainProtocol.v):
  15. drain_sequence_valid — Ok→Draining→Drained→Unmounting is valid
  16. drain_sequence_blocks_placement — no placement at any drain stage
  17. drain_then_unmount_safe — empty segment: drain + complete = safe to unmount
  18. drainable_no_processing — drainable segments have no in-progress transfers

  Layer 3 — Transfer-replica bridge (ReplicaTransfer.v):
  19. transfer_success_valid_transition — task succeeded → PROCESSING→COMPLETE valid
  20. transfer_failure_valid_transition — task failed → PROCESSING→FAILED valid
  21. removed_not_coupled_active — REMOVED replicas can't have active transfers

  Layer 4 — Composition (StoreComposition.v):
  22. four_gates_hold — all four protocol gates hold simultaneously on consistent store
  23. evictable_characterization — evictable ↔ complete ∧ not hard-pinned ∧ no readers
  24. processing_only_on_ok — PROCESSING objects exist only on SegOk segments
  25. place_preserves_consistency — placing an object preserves store consistency
  26. remove_preserves_consistency — removing an object preserves store consistency
  27. begin_drain_preserves_consistency — starting drain preserves store consistency
  28. no_incomplete_reads — no reads from non-COMPLETE replicas
  29. no_unmount_with_objects — unmount requires empty segment

KNOWN GAPS
==========

  - Eviction watermark policy: source triggers eviction when usage exceeds
    high_watermark_ratio (default 0.95).  We model eviction protection but
    not the trigger condition.
    STATUS: deferred.  REASON: requires rational arithmetic or fixed-point
    encoding; safety properties don't depend on the threshold value.

  - Lease-based pinning: source uses TTL leases (etcd) to temporarily pin
    objects.  We model pin levels but not time-based expiry.
    STATUS: deferred.  REASON: temporal properties need a clock model;
    the structural invariant (pinned → protected) holds regardless.

  - Concurrent segment access: source uses ScopedSegmentAccess with mutexes
    and atomic refcounts.  We model the logical invariants, not the locking.
    STATUS: abandoned.  REASON: Coq is not the right tool for lock-level
    verification; the value is in protocol composition, not mutex correctness.

  - Standby/HA promotion protocol: source has a 10-state machine
    (STOPPED→CONNECTING→SYNCING→WATCHING→PROMOTING→PROMOTED + error states).
    STATUS: deferred.  REASON: complex but orthogonal to KV cache correctness.
    Could be its own module if someone wants to formalize HA failover.

  - Replica variant types: source distinguishes MEMORY, DISK, LOCAL_DISK
    replicas with different descriptors (MemoryDescriptor, DiskDescriptor).
    We use a single ReplicaStatus without type discrimination.
    STATUS: deferred.  REASON: the lifecycle state machine is the same for
    all replica types; the type affects transport selection, not safety.

  - P2P shard replication: the Go-based p2p-store tracks Gold vs Replica
    locations per shard with etcd metadata.  This is a distribution protocol
    with its own consistency requirements.
    STATUS: deferred.  REASON: separate language (Go), separate concern
    (distribution vs lifecycle), would need its own formalization.

FORMALIZATION MAP
=================

Source module -> Coq module -> What and why.

  mooncake-store (replica.h, segment.h, types.h, eviction_strategy.h)
    → Core/Core.v
    FIDELITY: specification
    WHAT: Replica lifecycle (5-state machine), segment health (4-state
    machine), pin levels with ordering, object metadata with eviction
    protection predicate, transfer task with slice counting.
    WHY: These are the protocol-level types that all higher modules
    depend on.  Getting the state machines right prevents composition bugs.
    KEY RESULTS: replica_no_self_transition, no_transition_from_removed,
    processing_not_evictable, hard_pinned_protected, active_readers_protected

  mooncake-store (store.cpp, allocator.h)
    → Store/SegmentManager.v
    FIDELITY: specification
    WHAT: Segment bookkeeping (capacity tracking), object placement
    and removal, segment status transitions, drain/unmount safety
    preconditions.
    WHY: Placement on wrong-status segments and premature unmounting
    are the primary segment-level safety violations.
    KEY RESULTS: place_requires_ok, empty_drained_safe, place_preserves_status

  mooncake-transfer-engine (transport.h, transfer_task.h)
    → Transfer/TransferProtocol.v
    FIDELITY: specification
    WHAT: Transfer task state machine (4-state), slice counting with
    monotonicity invariant, completion coherence, task lifecycle operations.
    WHY: Transfer completion must be monotonic (slices only increase)
    and coherent (completed iff all slices done).  Terminal states must
    be absorbing.
    KEY RESULTS: create_task_slice_inv, complete_slice_preserves_inv,
    transfer_no_self_transition, task_finished_iff

  mooncake-store (segment.h drain logic, PrepareUnmount, CommitUnmount)
    → Store/DrainProtocol.v
    FIDELITY: specification
    WHAT: Two-phase drain/unmount protocol.  DRAINING blocks allocation,
    objects must be explicitly removed, DRAINED + empty = safe to unmount.
    WHY: Premature unmounting loses data.  The drain protocol is the
    primary defense; it must be proven correct as a sequence.
    KEY RESULTS: drain_sequence_valid, drain_sequence_blocks_placement,
    drain_then_unmount_safe, drainable_no_processing

  mooncake-store (replica.h mark_complete, transfer_task.h callback)
    → Transfer/ReplicaTransfer.v
    FIDELITY: specification
    WHAT: Bridge between transfer completion and replica status.
    task_succeeded → PROCESSING → COMPLETE.  task_failed → PROCESSING →
    FAILED.  REMOVED replicas cannot have active transfers.
    WHY: This is the critical coupling between transfer and replica
    state machines.  An orphaned transfer or premature promotion is
    a data integrity bug.
    KEY RESULTS: transfer_success_valid_transition,
    transfer_failure_valid_transition, removed_not_coupled_active

  composition of replica.h × segment.h × transfer_task.h
    → Composition/StoreComposition.v
    FIDELITY: specification
    WHAT: Composed global invariant (store_consistent) over segment +
    transfers.  Four protocol gates hold simultaneously.  Placement,
    removal, and drain operations preserve consistency.
    WHY: The capstone — this is what the formalization exists to prove.
    Individual state machines are trivial; their composition is where
    bugs hide.
    KEY RESULTS: four_gates_hold, evictable_characterization,
    processing_only_on_ok, place_preserves_consistency,
    remove_preserves_consistency, begin_drain_preserves_consistency,
    no_incomplete_reads, no_unmount_with_objects

NEXT STEPS
==========

  Items 1-3 from the original plan are DONE (StoreComposition,
  ReplicaTransfer, DrainProtocol).  Remaining extensions:

  1. EvictionSafety.v — Formalize the eviction precondition.
     Theorem: eviction of a protected object requires explicit unpin
     AND ref_count = 0 AND status = COMPLETE.
     Theorem: eviction removes exactly one object and decreases used space.
     Theorem: eviction on a draining segment does not create new replicas.
     Source: eviction_strategy.h EvictKey, store.cpp eviction loop.

  2. CapacityCoherence.v — Prove segment capacity bookkeeping is coherent.
     Theorem: place increases used by om_size, remove decreases by om_size.
     Theorem: if has_room checked before place, then used ≤ capacity after.
     Theorem: used = sum of om_size over all live objects on the segment.
     Source: allocator.h, store.cpp allocation path.

  3. RefcountProtocol.v — Formalize reference counting as a protocol.
     Theorem: inc_refcnt/dec_refcnt are paired (every inc has a matching dec).
     Theorem: refcnt > 0 → eviction_protected.
     Theorem: refcnt = 0 is a precondition for replica removal.
     Source: replica.h inc_refcnt/dec_refcnt/is_busy.
