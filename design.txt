Mooncake KV cache disaggregated serving

What we formalize and why:

  Mooncake disaggregates KV cache from the inference engine: prefill
  nodes produce KV blocks stored in a distributed pool, decode nodes
  fetch them on demand.  The core correctness question is whether the
  replica lifecycle, segment health, eviction, and transfer protocols
  compose safely — no reads from incomplete replicas, no eviction of
  pinned or in-flight data, no allocation on draining segments, no
  transfer outliving its replica.  Formal verification catches state
  machine composition bugs (e.g. draining a segment with in-progress
  writes) that unit tests miss because they don't explore all interleavings.

  We prove: (1) replica lifecycle monotonicity and readability gates,
  (2) segment drain/unmount safety, (3) eviction protection coherence,
  (4) transfer slice monotonicity and completion coherence.

What we model and what we leave out:

  MODEL (high verification value):
  - Replica status state machine (INITIALIZED→PROCESSING→COMPLETE→REMOVED, +FAILED)
  - Segment status lifecycle (OK→DRAINING→DRAINED→UNMOUNTING)
  - Eviction protection (pin levels, ref counting, status gates)
  - Transfer task lifecycle (PENDING→ACTIVE→COMPLETED/FAILED)
  - Slice counting invariant (done_slices ≤ total_slices, monotonic)
  - Segment capacity bookkeeping (used ≤ capacity)
  - Object metadata and placement safety (allocate only on SegOk)
  - Drain/unmount preconditions (empty + drained before unmount)

  LEAVE OUT (low verification value or too coupled to internals):
  - RDMA/NVLink transport layer details (hardware-specific, not protocol-level)
  - Allocator implementations (CachelibBuffer, OffsetBuffer — memory management internals)
  - Standby/HA state machine (STOPPED→CONNECTING→...→PROMOTED — complex but orthogonal)
  - OpLog replication and snapshot mechanics (persistence layer, not core protocol)
  - P2P store shard distribution (Go codebase, etcd metadata — different language/paradigm)
  - EP (expert parallelism) — separate concern, formalized in deepep_formal
  - File storage backend, io_uring — I/O implementation details
  - Eviction strategy implementations (LRU/FIFO ordering — algorithm, not safety)
  - Lease TTL renewal mechanics (timing-dependent, hard to model usefully)
  - IBGDA/RoCE queue pair setup (hardware initialization)

MODELING DECISIONS
==================

Non-obvious choices that affect multiple files.

  1. Replica status without UNDEFINED: Source has 6 states (UNDEFINED,
     INITIALIZED, PROCESSING, COMPLETE, FAILED, REMOVED).  We drop
     UNDEFINED — it represents uninitialized memory, not a protocol
     state.  A replica enters our model at INITIALIZED.
     WHY: UNDEFINED has no meaningful transitions and adds noise to
     every exhaustive case analysis.

  2. Weights as nat, not floats: Eviction priority, transfer progress,
     and object sizes are all nat.  We don't model floating-point
     watermark ratios (e.g. high_watermark_ratio=0.95).
     WHY: The safety properties (eviction protection, capacity bounds)
     are structural, not numerical.  Nat arithmetic keeps proofs
     decidable.

  3. Pin levels as a 3-element type, not nat: Source uses implicit
     priority; we make it explicit with Unpinned < SoftPinned < HardPinned
     and a decidable ordering.
     WHY: Makes eviction protection predicates compositional — we can
     state "hard-pinned implies protected" as a standalone lemma.

  4. Transfer slices as counts, not buffers: Source tracks actual data
     buffers per slice with variant descriptors (RDMA, memcpy, file).
     We track only (done_slices, total_slices) counts.
     WHY: The safety property is monotonicity (done ≤ total, done
     increases), not data content.  Buffer types are transport-layer
     concerns we exclude.

  5. Segment status without UNDEFINED: Same rationale as replica — a
     segment enters our model at SegOk (mounted and available).
     WHY: Keeps segment_transition inductive clean.

  6. Eviction protection is a bool predicate, not a lock: Source uses
     atomic refcnt + weak_ptr + scoped access.  We model the logical
     predicate: protected iff hard-pinned OR ref_count > 0 OR not-yet-
     readable.
     WHY: The invariant is "protected objects are not evicted", not
     "the lock is held correctly".  The latter is a concurrency concern
     we don't model.

HEADLINE THEOREMS
=================

  1. replica_no_self_transition (Core.v) — no replica status is its own successor
  2. segment_no_self_transition (Core.v) — no segment status is its own successor
  3. no_transition_from_removed (Core.v) — REMOVED is terminal for replicas
  4. no_transition_from_unmounting (Core.v) — UNMOUNTING is terminal for segments
  5. processing_not_evictable (Core.v) — in-progress writes are always protected
  6. hard_pinned_protected (Core.v) — hard-pinned objects are always protected
  7. active_readers_protected (Core.v) — objects with ref_count > 0 are protected
  8. readable_is_complete (Core.v) — readability implies COMPLETE status
  9. place_requires_ok (SegmentManager.v) — placement requires SegOk segment
  10. empty_drained_safe (SegmentManager.v) — empty + drained = safe to unmount
  11. create_task_slice_inv (TransferProtocol.v) — new tasks satisfy slice invariant
  12. complete_slice_preserves_inv (TransferProtocol.v) — slice completion is monotonic
  13. transfer_no_self_transition (TransferProtocol.v) — no transfer status loops
  14. task_finished_iff (TransferProtocol.v) — task_finished reflects terminal status

KNOWN GAPS
==========

  - Eviction watermark policy: source triggers eviction when usage exceeds
    high_watermark_ratio (default 0.95).  We model eviction protection but
    not the trigger condition.
    STATUS: deferred.  REASON: requires rational arithmetic or fixed-point
    encoding; safety properties don't depend on the threshold value.

  - Lease-based pinning: source uses TTL leases (etcd) to temporarily pin
    objects.  We model pin levels but not time-based expiry.
    STATUS: deferred.  REASON: temporal properties need a clock model;
    the structural invariant (pinned → protected) holds regardless.

  - Concurrent segment access: source uses ScopedSegmentAccess with mutexes
    and atomic refcounts.  We model the logical invariants, not the locking.
    STATUS: abandoned.  REASON: Coq is not the right tool for lock-level
    verification; the value is in protocol composition, not mutex correctness.

  - Standby/HA promotion protocol: source has a 10-state machine
    (STOPPED→CONNECTING→SYNCING→WATCHING→PROMOTING→PROMOTED + error states).
    STATUS: deferred.  REASON: complex but orthogonal to KV cache correctness.
    Could be its own module if someone wants to formalize HA failover.

  - Replica variant types: source distinguishes MEMORY, DISK, LOCAL_DISK
    replicas with different descriptors (MemoryDescriptor, DiskDescriptor).
    We use a single ReplicaStatus without type discrimination.
    STATUS: deferred.  REASON: the lifecycle state machine is the same for
    all replica types; the type affects transport selection, not safety.

  - P2P shard replication: the Go-based p2p-store tracks Gold vs Replica
    locations per shard with etcd metadata.  This is a distribution protocol
    with its own consistency requirements.
    STATUS: deferred.  REASON: separate language (Go), separate concern
    (distribution vs lifecycle), would need its own formalization.

FORMALIZATION MAP
=================

Source module -> Coq module -> What and why.

  mooncake-store (replica.h, segment.h, types.h, eviction_strategy.h)
    → Core/Core.v
    FIDELITY: specification
    WHAT: Replica lifecycle (5-state machine), segment health (4-state
    machine), pin levels with ordering, object metadata with eviction
    protection predicate, transfer task with slice counting.
    WHY: These are the protocol-level types that all higher modules
    depend on.  Getting the state machines right prevents composition bugs.
    KEY RESULTS: replica_no_self_transition, no_transition_from_removed,
    processing_not_evictable, hard_pinned_protected, active_readers_protected

  mooncake-store (store.cpp, allocator.h)
    → Store/SegmentManager.v
    FIDELITY: specification
    WHAT: Segment bookkeeping (capacity tracking), object placement
    and removal, segment status transitions, drain/unmount safety
    preconditions.
    WHY: Placement on wrong-status segments and premature unmounting
    are the primary segment-level safety violations.
    KEY RESULTS: place_requires_ok, empty_drained_safe, place_preserves_status

  mooncake-transfer-engine (transport.h, transfer_task.h)
    → Transfer/TransferProtocol.v
    FIDELITY: specification
    WHAT: Transfer task state machine (4-state), slice counting with
    monotonicity invariant, completion coherence, task lifecycle operations.
    WHY: Transfer completion must be monotonic (slices only increase)
    and coherent (completed iff all slices done).  Terminal states must
    be absorbing.
    KEY RESULTS: create_task_slice_inv, complete_slice_preserves_inv,
    transfer_no_self_transition, task_finished_iff

NEXT STEPS
==========

  Possible extensions, roughly in priority order:

  1. EvictionSafety.v — Compose eviction_protected with segment operations.
     Theorem: eviction of a protected object requires explicit unpin first.
     Source: eviction_strategy.h, store.cpp eviction loop.

  2. ReplicaTransfer.v — Link transfer completion to replica status
     transitions.  Theorem: task_succeeded → replica can transition
     PROCESSING → COMPLETE.  Source: store.cpp callback on transfer done.

  3. DrainProtocol.v — Formalize the full drain sequence: mark DRAINING,
     migrate all replicas, verify empty, transition to DRAINED, unmount.
     Theorem: drain preserves no-data-loss (all replicas migrated before
     unmount).  Source: segment.h drain logic.

  4. CapacityCoherence.v — Prove that segment used-space tracking is
     coherent: place increases by om_size, remove decreases by om_size,
     used never exceeds capacity if has_room checked.
     Source: allocator.h, store.cpp allocation path.
