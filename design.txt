Mooncake KV cache disaggregated serving

What we formalize and why:

  Mooncake disaggregates KV cache from GPU compute in LLM serving.
  Prefill nodes produce KV blocks; these are stored in a distributed
  pool (DRAM, SSD, CXL); decode nodes fetch them on demand via RDMA.
  The core insight: trade cheap storage for expensive GPU recomputation.

  Strip away the transport layer and the scheduler — what remains is a
  distributed object store with a lifecycle protocol.  The objects are
  KV cache blocks.  The lifecycle is: produce → transfer → store →
  serve → evict.  The store manages two interlocking state machines:
  replica status (per-object) and segment status (per-container).

  The core correctness question: do the protocol gates compose safely?
  Each gate is trivial alone — readable iff COMPLETE, allocatable iff
  SegOk, protected iff pinned or referenced.  But their COMPOSITION
  has subtle interleavings that testing can't enumerate: can you drain
  a segment while a transfer is writing to a replica on it?  Can you
  evict an object that another transfer is about to read?

  We prove: (1) each state machine has no self-transitions and terminal
  states are absorbing, (2) eviction protection is coherent (pinned OR
  referenced OR in-progress → protected), (3) segment operations respect
  status gates, (4) transfer slice counting is monotonic, (5) the
  protocol gates compose into a single global invariant.

What we model and what we leave out:

  MODEL (high verification value):
  - Replica status state machine (INITIALIZED→PROCESSING→COMPLETE→REMOVED, +FAILED)
  - Segment status lifecycle (OK→DRAINING→DRAINED→UNMOUNTING)
  - Eviction protection (pin levels, ref counting, status gates)
  - Transfer task lifecycle (PENDING→ACTIVE→COMPLETED/FAILED)
  - Slice counting invariant (done_slices ≤ total_slices, monotonic)
  - Segment capacity bookkeeping (used ≤ capacity)
  - Object metadata and placement safety (allocate only on SegOk)
  - Drain/unmount preconditions (empty + drained before unmount)

  LEAVE OUT (low verification value or too coupled to internals):
  - RDMA/NVLink transport layer details (hardware-specific, not protocol-level)
  - Allocator implementations (CachelibBuffer, OffsetBuffer — memory management internals)
  - Standby/HA state machine (STOPPED→CONNECTING→...→PROMOTED — complex but orthogonal)
  - OpLog replication and snapshot mechanics (persistence layer, not core protocol)
  - P2P store shard distribution (Go codebase, etcd metadata — different language/paradigm)
  - EP (expert parallelism) — separate concern, formalized in deepep_formal
  - File storage backend, io_uring — I/O implementation details
  - Eviction strategy implementations (LRU/FIFO ordering — algorithm, not safety)
  - Lease TTL renewal mechanics (timing-dependent, hard to model usefully)
  - IBGDA/RoCE queue pair setup (hardware initialization)

MODELING DECISIONS
==================

Non-obvious choices that affect multiple files.

  1. Replica status without UNDEFINED: Source has 6 states (UNDEFINED,
     INITIALIZED, PROCESSING, COMPLETE, FAILED, REMOVED).  We drop
     UNDEFINED — it represents uninitialized memory, not a protocol
     state.  A replica enters our model at INITIALIZED.
     WHY: UNDEFINED has no meaningful transitions and adds noise to
     every exhaustive case analysis.

  2. Weights as nat, not floats: Eviction priority, transfer progress,
     and object sizes are all nat.  We don't model floating-point
     watermark ratios (e.g. high_watermark_ratio=0.95).
     WHY: The safety properties (eviction protection, capacity bounds)
     are structural, not numerical.  Nat arithmetic keeps proofs
     decidable.

  3. Pin levels as a 3-element type, not nat: Source uses implicit
     priority; we make it explicit with Unpinned < SoftPinned < HardPinned
     and a decidable ordering.
     WHY: Makes eviction protection predicates compositional — we can
     state "hard-pinned implies protected" as a standalone lemma.

  4. Transfer slices as counts, not buffers: Source tracks actual data
     buffers per slice with variant descriptors (RDMA, memcpy, file).
     We track only (done_slices, total_slices) counts.
     WHY: The safety property is monotonicity (done ≤ total, done
     increases), not data content.  Buffer types are transport-layer
     concerns we exclude.

  5. Segment status without UNDEFINED: Same rationale as replica — a
     segment enters our model at SegOk (mounted and available).
     WHY: Keeps segment_transition inductive clean.

  6. Eviction protection is a bool predicate, not a lock: Source uses
     atomic refcnt + weak_ptr + scoped access.  We model the logical
     predicate: protected iff hard-pinned OR ref_count > 0 OR not-yet-
     readable.
     WHY: The invariant is "protected objects are not evicted", not
     "the lock is held correctly".  The latter is a concurrency concern
     we don't model.

  7. LTS labels are 5 operations, not 9: The initial plan called for 9
     labels (including transfer operations).  We use 5: LPlace, LRemove,
     LBeginDrain, LCompleteDrain, LUnmount.  Transfer operations are
     excluded from the LTS because the store_step relation operates on
     StoreState (segment + transfers), and transfer status changes don't
     interact with segment operations in the current model.
     WHY: Keeping the LTS minimal simplifies case analysis in every
     theorem.  Transfer properties are proved separately in
     TransferProtocol.v.

  8. Drain liveness uses object_count as Lyapunov function, not
     composed_progress_sound: The plan called for instantiating DML's
     composed progress framework.  Instead, we use a simpler argument:
     object_count is bounded below by 0 and never increases on draining
     segments, so the drain phase terminates in at most object_count steps.
     WHY: The composed progress framework requires an active/frame
     decomposition with explicit per-component indices.  The Lyapunov
     argument is self-contained and sufficient for drain termination.

  10. Enriched LTS as a separate relation: The read/evict operations
     are defined in a separate rich_store_step relation rather than
     extending store_step.  This avoids breaking all existing proofs
     (which would need more cases) while still enabling the safety
     theorems.
     WHY: Extending store_step with 3 new constructors would require
     updating every proof that does case analysis on store_step
     (store_step_preserves_consistency, drain_never_adds_objects, etc.).
     The enriched LTS duplicates the 5 original constructors but keeps
     existing theorems intact.

  11. Evict vs Remove: The existing LRemove is unconditional (used during
     drain cleanup).  RLEvict adds preconditions (eviction_protected =
     false).  In the real system, eviction and drain-cleanup are distinct
     code paths with different safety checks.
     WHY: The distinction captures the real protocol: eviction is guarded
     by refcount/pin/status checks, while drain-time removal is forced
     (the segment is being reclaimed regardless).

  9. drain_wf (well-foundedness of drain_step) is not provable: The
     drain_step relation wraps LRemove, but remove_object is always
     enabled (it's a filter that may be a no-op).  A no-op removal
     doesn't decrease any measure, so rank_wf cannot be instantiated.
     WHY: Well-foundedness requires every step to make progress, but
     removal of a non-existent object is a valid step that does nothing.
     The bounded theorem (drain_bounded) captures the useful property
     without requiring well-foundedness.

HEADLINE THEOREMS
=================

  Layer 1 — State machines (Core.v):
  1. replica_no_self_transition — no replica status is its own successor
  2. segment_no_self_transition — no segment status is its own successor
  3. no_transition_from_removed — REMOVED is terminal for replicas
  4. no_transition_from_unmounting — UNMOUNTING is terminal for segments
  5. processing_not_evictable — in-progress writes are always protected
  6. hard_pinned_protected — hard-pinned objects are always protected
  7. active_readers_protected — objects with ref_count > 0 are protected
  8. readable_is_complete — readability implies COMPLETE status

  Layer 2 — Segment operations (SegmentManager.v):
  9. place_requires_ok — placement requires SegOk segment
  10. empty_drained_safe — empty + drained = safe to unmount

  Layer 2 — Transfer protocol (TransferProtocol.v):
  11. create_task_slice_inv — new tasks satisfy slice invariant
  12. complete_slice_preserves_inv — slice completion is monotonic
  13. transfer_no_self_transition — no transfer status loops
  14. task_finished_iff — task_finished reflects terminal status

  Layer 2 — Unified LTS (StoreLTS.v):
  15. store_step_preserves_consistency — every store_step preserves store_consistent
  16. store_consistent_star — store_consistent through arbitrary step sequences (via DML lstar)

  Layer 2-3 — Acyclicity and well-foundedness (Acyclicity.v):
  17. replica_acyclic — no replica status reaches itself through any path (subsumes #1)
  18. segment_acyclic — no segment status reaches itself through any path (subsumes #2)
  19. transfer_acyclic — no transfer status reaches itself through any path
  20. replica_wf, segment_wf, transfer_wf — all component transitions are well-founded
  21. progress_step_rank_nonincreasing — progress steps never increase composed rank
  22. drain_step_decreases_rank — drain operations strictly decrease composed rank

  Layer 3 — Drain protocol (DrainProtocol.v):
  23. drain_sequence_valid — Ok→Draining→Drained→Unmounting is valid
  24. drain_sequence_blocks_placement — no placement at any drain stage
  25. drain_then_unmount_safe — empty segment: drain + complete = safe to unmount
  26. drainable_no_processing — drainable segments have no in-progress transfers

  Layer 3 — Transfer-replica bridge (ReplicaTransfer.v):
  27. transfer_success_valid_transition — task succeeded → PROCESSING→COMPLETE valid
  28. transfer_failure_valid_transition — task failed → PROCESSING→FAILED valid
  29. removed_not_coupled_active — REMOVED replicas can't have active transfers

  Layer 3 — Refinement morphisms (StoreRefinement.v):
  30. segment_invariant_lift — component invariants lift to composed store (via DML Refinement)
  31. segment_trace_projection — store traces project to valid segment traces
  32. once_draining_never_ok — draining segment never returns to SegOk
  33. once_draining_no_placement — once draining, placement is always blocked
  34. segment_rank_monotone — segment rank never increases through store traces

  Layer 4 — Composition (StoreComposition.v):
  35. four_gates_hold — all four protocol gates hold simultaneously on consistent store
  36. evictable_characterization — evictable ↔ complete ∧ not hard-pinned ∧ no readers
  37. processing_only_on_ok — PROCESSING objects exist only on SegOk segments
  38. place_preserves_consistency — placing an object preserves store consistency
  39. remove_preserves_consistency — removing an object preserves store consistency
  40. begin_drain_preserves_consistency — starting drain preserves store consistency
  41. no_incomplete_reads — no reads from non-COMPLETE replicas
  42. no_unmount_with_objects — unmount requires empty segment

  Layer 4 — Drain liveness (DrainProgress.v):
  43. drain_never_adds_objects — object count never increases on non-allocatable segments
  44. remove_present_decreases — removing a present object strictly decreases count
  45. drain_completion_enabled — empty draining segment enables CompleteDrain
  46. drain_trace_monotone — object count never increases through removal traces
  47. drain_invariant — draining status and consistency preserved through removal traces
  48. drain_bounded — drain phase bounded by initial object count; emptiness enables completion

  Layer 4 — Lifecycle convergence (Convergence.v):
  49. filter_comm — filter operations commute (order-independence of removals)
  50. remove_objects_comm — removal order doesn't affect resulting objects list
  51. sequential_remove_lstar — removing all objects one by one empties the segment
  52. lifecycle_convergence — HEADLINE: segment reaches SegUnmounting in exactly n+3 steps
  53. lifecycle_trace_length — tight bound: trace length = object_count + 3
  54. segment_reclaimable — every segment with unique IDs is reclaimable

  Layer 2-3 — Refcount protocol (RefcountProtocol.v):
  55. inc_protects — incrementing refcount makes object eviction-protected
  56. evictable_zero_refcnt — evictable implies refcount = 0
  57. evictable_is_complete — evictable implies status = COMPLETE
  58. inc_segment_refcnt_protects — inc on segment produces protected object
  59. find_after_update — object lookup after update_object_meta gives f(original)
  60. update_preserves_live — refcount changes preserve object liveness

  Layer 4 — Eviction safety (EvictionSafety.v):
  61. read_eviction_safety — HEADLINE: refcnt > 0 prevents eviction
  62. begin_read_blocks_eviction — HEADLINE: BeginRead makes Evict impossible
  63. evict_is_guarded_remove — Evict is a precondition-checked Remove
  64. triple_protection — pin, refcount, or non-readable each independently protect
  65. eviction_requires_all_absent — eviction requires all three protections absent

KNOWN GAPS
==========

  - Eviction watermark policy: source triggers eviction when usage exceeds
    high_watermark_ratio (default 0.95).  We model eviction protection but
    not the trigger condition.
    STATUS: deferred.  REASON: requires rational arithmetic or fixed-point
    encoding; safety properties don't depend on the threshold value.

  - Lease-based pinning: source uses TTL leases (etcd) to temporarily pin
    objects.  We model pin levels but not time-based expiry.
    STATUS: deferred.  REASON: temporal properties need a clock model;
    the structural invariant (pinned → protected) holds regardless.

  - Concurrent segment access: source uses ScopedSegmentAccess with mutexes
    and atomic refcounts.  We model the logical invariants, not the locking.
    STATUS: abandoned.  REASON: Coq is not the right tool for lock-level
    verification; the value is in protocol composition, not mutex correctness.

  - Standby/HA promotion protocol: source has a 10-state machine
    (STOPPED→CONNECTING→SYNCING→WATCHING→PROMOTING→PROMOTED + error states).
    STATUS: deferred.  REASON: complex but orthogonal to KV cache correctness.
    Could be its own module if someone wants to formalize HA failover.

  - Replica variant types: source distinguishes MEMORY, DISK, LOCAL_DISK
    replicas with different descriptors (MemoryDescriptor, DiskDescriptor).
    We use a single ReplicaStatus without type discrimination.
    STATUS: deferred.  REASON: the lifecycle state machine is the same for
    all replica types; the type affects transport selection, not safety.

  - P2P shard replication: the Go-based p2p-store tracks Gold vs Replica
    locations per shard with etcd metadata.  This is a distribution protocol
    with its own consistency requirements.
    STATUS: deferred.  REASON: separate language (Go), separate concern
    (distribution vs lifecycle), would need its own formalization.

  - drain_wf (well-foundedness of drain_step relation): drain_step wraps
    LRemove, but remove_object is always enabled as a no-op filter.
    Removing a non-existent object doesn't decrease any measure.
    STATUS: abandoned.  REASON: rank_wf requires strict decrease on every
    step.  The useful property (drain terminates after bounded removals)
    is captured by drain_bounded without needing well-foundedness.

  - Transfer operations in the LTS: the plan called for 9 store labels
    including LBeginTransfer, LCompleteSlice, LFinishTransfer,
    LPromoteReplica.  Only 5 labels were implemented.
    STATUS: deferred.  REASON: transfer status changes don't interact
    with segment operations in the current model.  Adding them would
    increase case analysis in every theorem without enabling new
    cross-cutting results.  Can be added when transfer-segment coupling
    properties are needed.

  - Replica refinement morphism: StoreRefinement.v was planned to include
    both segment and replica projections.  Only segment refinement was
    implemented.
    STATUS: deferred.  REASON: replica projection requires a find-by-id
    function and handling of absent objects (project to RsRemoved).
    Segment refinement was higher priority since it enables drain
    monotonicity.  Can be added when per-object trace reasoning is needed.

FORMALIZATION MAP
=================

Source module -> Coq module -> What and why.

  mooncake-store (replica.h, segment.h, types.h, eviction_strategy.h)
    → Core/Core.v
    FIDELITY: specification
    WHAT: Replica lifecycle (5-state machine), segment health (4-state
    machine), pin levels with ordering, object metadata with eviction
    protection predicate, transfer task with slice counting.
    WHY: These are the protocol-level types that all higher modules
    depend on.  Getting the state machines right prevents composition bugs.
    KEY RESULTS: replica_no_self_transition, no_transition_from_removed,
    processing_not_evictable, hard_pinned_protected, active_readers_protected

  mooncake-store (store.cpp, allocator.h)
    → Store/SegmentManager.v
    FIDELITY: specification
    WHAT: Segment bookkeeping (capacity tracking), object placement
    and removal, segment status transitions, drain/unmount safety
    preconditions.
    WHY: Placement on wrong-status segments and premature unmounting
    are the primary segment-level safety violations.
    KEY RESULTS: place_requires_ok, empty_drained_safe, place_preserves_status

  mooncake-transfer-engine (transport.h, transfer_task.h)
    → Transfer/TransferProtocol.v
    FIDELITY: specification
    WHAT: Transfer task state machine (4-state), slice counting with
    monotonicity invariant, completion coherence, task lifecycle operations.
    WHY: Transfer completion must be monotonic (slices only increase)
    and coherent (completed iff all slices done).  Terminal states must
    be absorbing.
    KEY RESULTS: create_task_slice_inv, complete_slice_preserves_inv,
    transfer_no_self_transition, task_finished_iff

  mooncake-store (segment.h drain logic, PrepareUnmount, CommitUnmount)
    → Store/DrainProtocol.v
    FIDELITY: specification
    WHAT: Two-phase drain/unmount protocol.  DRAINING blocks allocation,
    objects must be explicitly removed, DRAINED + empty = safe to unmount.
    WHY: Premature unmounting loses data.  The drain protocol is the
    primary defense; it must be proven correct as a sequence.
    KEY RESULTS: drain_sequence_valid, drain_sequence_blocks_placement,
    drain_then_unmount_safe, drainable_no_processing

  mooncake-store (replica.h mark_complete, transfer_task.h callback)
    → Transfer/ReplicaTransfer.v
    FIDELITY: specification
    WHAT: Bridge between transfer completion and replica status.
    task_succeeded → PROCESSING → COMPLETE.  task_failed → PROCESSING →
    FAILED.  REMOVED replicas cannot have active transfers.
    WHY: This is the critical coupling between transfer and replica
    state machines.  An orphaned transfer or premature promotion is
    a data integrity bug.
    KEY RESULTS: transfer_success_valid_transition,
    transfer_failure_valid_transition, removed_not_coupled_active

  composition of replica.h × segment.h × transfer_task.h
    → Composition/StoreComposition.v
    FIDELITY: specification
    WHAT: Composed global invariant (store_consistent) over segment +
    transfers.  Four protocol gates hold simultaneously.  Placement,
    removal, and drain operations preserve consistency.
    WHY: The capstone — this is what the formalization exists to prove.
    Individual state machines are trivial; their composition is where
    bugs hide.
    KEY RESULTS: four_gates_hold, evictable_characterization,
    processing_only_on_ok, place_preserves_consistency,
    remove_preserves_consistency, begin_drain_preserves_consistency,
    no_incomplete_reads, no_unmount_with_objects

  composition of replica.h × segment.h × transfer_task.h
    → LTS/StoreLTS.v
    FIDELITY: specification
    WHAT: Unified labeled transition system.  5 labels (LPlace, LRemove,
    LBeginDrain, LCompleteDrain, LUnmount).  store_step wraps existing
    operations with precondition guards.  Multi-step preservation via DML
    lstar_preserved.
    WHY: Enables multi-step reasoning about arbitrary execution traces.
    Single-step preservation is necessary but not sufficient for safety
    under interleaving.
    KEY RESULTS: store_step_preserves_consistency, store_consistent_star,
    store_step_segment_status

  composition of replica.h × segment.h × transfer_task.h
    → LTS/Acyclicity.v
    FIDELITY: specification
    WHAT: Rank functions for all three component state machines.  Global
    acyclicity via DML rank_acyclic.  Well-foundedness via DML rank_wf.
    Composed store_rank with progress step analysis.
    WHY: Acyclicity subsumes the single-step no-self-transition lemmas
    (proves the stronger property: no state reaches itself via any path
    length).  Well-foundedness guarantees convergence without external input.
    KEY RESULTS: replica_acyclic, segment_acyclic, transfer_acyclic,
    replica_wf, segment_wf, transfer_wf, progress_step_rank_nonincreasing,
    drain_step_decreases_rank

  composition of replica.h × segment.h
    → LTS/StoreRefinement.v
    FIDELITY: specification
    WHAT: Refinement morphism projecting store state to segment component.
    Invariant lifting and trace projection via DML Refinement module.
    Segment monotonicity: status never regresses.
    WHY: Enables reasoning about component-level properties from composed
    traces.  Monotonicity proves that drain is irreversible: once draining
    begins, the segment can never accept new objects.
    KEY RESULTS: segment_invariant_lift, segment_trace_projection,
    once_draining_never_ok, once_draining_no_placement,
    segment_rank_monotone

  mooncake-store/include/segment.h (drain lifecycle)
    → LTS/DrainProgress.v
    FIDELITY: specification
    WHAT: Drain termination as a liveness property.  Object count is a
    Lyapunov function: bounded below by 0, never increases on draining
    segments, strictly decreases on removal of present objects.
    WHY: First liveness property in the formalization.  Safety says "bad
    things don't happen"; liveness says "good things eventually happen".
    Drain termination means a segment can eventually be reclaimed.
    KEY RESULTS: drain_never_adds_objects, remove_present_decreases,
    drain_completion_enabled, drain_trace_monotone, drain_invariant,
    drain_bounded

  mooncake-store/include/segment.h (full lifecycle)
    → LTS/Convergence.v
    FIDELITY: specification
    WHAT: Tight quantitative lifecycle bound.  Constructs the explicit
    optimal trace [BeginDrain, Remove₁…Removeₙ, CompleteDrain, Unmount]
    and proves it reaches SegUnmounting in exactly n+3 steps.  Also
    proves removal commutativity (order doesn't matter).
    WHY: Quantitative result — not just "the system converges" but
    "it converges in exactly this many steps via this specific trace."
    The commutativity result reveals inherent parallelism in the drain
    phase: all n removals are independent and could execute concurrently.
    KEY RESULTS: filter_comm, remove_objects_comm, sequential_remove_lstar,
    lifecycle_convergence, lifecycle_trace_length, segment_reclaimable

  mooncake-store/include/replica.h (inc_refcnt, dec_refcnt, fn_is_busy)
    → Store/RefcountProtocol.v
    FIDELITY: specification
    WHAT: Reference counting operations on objects and segments.  Object
    lookup by ID, in-place metadata update, refcount increment/decrement.
    Connection between refcount and eviction protection predicate.
    WHY: The refcount mechanism is the primary safety mechanism in
    Mooncake — it prevents eviction of objects being read via RDMA.
    Formalizing the operations and their properties is prerequisite for
    the eviction safety theorem.
    KEY RESULTS: inc_protects, evictable_zero_refcnt, evictable_is_complete,
    inc_segment_refcnt_protects, find_after_update, update_preserves_live

  mooncake-store/include/replica.h + src/master_service.cpp
    → LTS/EvictionSafety.v
    FIDELITY: specification
    WHAT: Enriched LTS with 8 operations (original 5 + BeginRead, EndRead,
    Evict).  BeginRead increments refcount on COMPLETE objects.  EndRead
    decrements.  Evict requires eviction_protected = false.  Key safety
    theorem: a read in progress (refcnt > 0) makes eviction impossible.
    WHY: This is the essence of Mooncake's safety.  The refcount protocol
    creates a separation between "active read" and "eligible for eviction"
    that cannot be violated by any interleaving of operations.
    KEY RESULTS: read_eviction_safety, begin_read_blocks_eviction,
    evict_is_guarded_remove, triple_protection, eviction_requires_all_absent

NEXT STEPS
==========

  Items 1-3 from the original plan are DONE (StoreComposition,
  ReplicaTransfer, DrainProtocol).  Remaining extensions:

  1. EvictionSafety.v — Formalize the eviction precondition.
     Theorem: eviction of a protected object requires explicit unpin
     AND ref_count = 0 AND status = COMPLETE.
     Theorem: eviction removes exactly one object and decreases used space.
     Theorem: eviction on a draining segment does not create new replicas.
     Source: eviction_strategy.h EvictKey, store.cpp eviction loop.

  2. CapacityCoherence.v — Prove segment capacity bookkeeping is coherent.
     Theorem: place increases used by om_size, remove decreases by om_size.
     Theorem: if has_room checked before place, then used ≤ capacity after.
     Theorem: used = sum of om_size over all live objects on the segment.
     Source: allocator.h, store.cpp allocation path.

  3. RefcountProtocol.v — Formalize reference counting as a protocol.
     Theorem: inc_refcnt/dec_refcnt are paired (every inc has a matching dec).
     Theorem: refcnt > 0 → eviction_protected.
     Theorem: refcnt = 0 is a precondition for replica removal.
     Source: replica.h inc_refcnt/dec_refcnt/is_busy.
